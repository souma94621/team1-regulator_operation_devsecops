# ТЗ
## ТЗ С8. Регулятор
### Базовые требования
БТ1. Система предполагает использование нескольких доменов безопасности, если особенности системы не предполагают иное (см. Брокер)

БТ2. Взаимодействие между доменами безопасности должно происходить только через брокер сообщений и проверяться монитором безопасности на соответствие политикам безопасности.

БТ3. Для системы должны быть перечислены и проанализированы активы, сформулированы цели и предположения безопасности, определены уровни доверия всех доменов безопасности.

БТ4. Для проверки политик безопасности должны быть реализованы автоматизированные тесты.

БТ5. Код доверенных доменов безопасности должен быть покрыт тестами не менее чем на 60%.

БТ6. Код доверенных доменов безопасности, повышающих целостность данных, должен быть покрыт тестами не менее чем на 70%, *включая зависимости*. 

Все зависимости должны быть проанализированы на предмет известных уязвимостей. В доверенных доменах безопасности запрещается использовать сторонние пакеты с известными уязвимостями.

Оговорка: т.к. проект учебный, в отдельных случаях допускаются исключения для тестирования зависимостей, в таком случае должен быть проведён статический анализ кода, проанализирована информация в публичных банках данных угроз (например, БДУ ФСТЭК)

БТ7. Код остальных (недоверенных) доменов безопасности должен быть протестирован не менее чем на 30% (без учёта зависимостей).

БТ8. Выдача целей безопасности по запросу в формате сертификата системы, подписанного Регулятором (если применимо)

БТ9. Выдача событий безопасности по запросу (если применимо)

БТ10. Передача телеметрии в сервис аналитики

БТ11. Передача событий безопасности в сервис аналитики

БТ12. Система (за исключением вспомогательных типа SITL) должна иметь сертификат безопасности с указанием целей безопасности (если применимо). Исключения допускаются для сторонних систем типа брокеров сообщений, баз данных и т.п. Сертификат безопасности должен быть подписан Регулятором / Авторизованным Сертифицирующим Центром (АСЦ)

БТ13. События должны записываться в журнал (с указанием уровня критичности)
### Основные функции
ОФ1. Обработка нового запроса на генерацию сертификата с заданными целями безопасности
- выполнение тестов безопасности
- контроль покрытия доверенного кода тестами
- фиксация целей безопасности в сертификате
ОФ2. Проверка подлинности сертификата по запросу
ОФ3. Отзыв сертификата 
ОФ4. Передача телеметрии в наземные системы в процессе полёта
ОФ5. Запрос на выплату страховки по инциденту с учётом
- ID заказа
- суммы ущерба

## ТЗ П1. DevOps
### Базовые требования
БТ1. В master ветке всегда должен находиться рабочий код и скрипты.
Как проверять: рабочее окружение должно настраиваться скриптами на чистой виртуалке без ошибок и все автотесты должны выполняться успешно
БТ2. Должны быть настроены автоматические проверки перед отправкой изменений в репозиторий (pre-commit hook).
Как проверять: перед отправкой изменений должны автоматически выполняться юнит-тесты. В случае ошибок тестирования отправка изменений должна блокироваться
БТ3. Все скрипты для настройки инфраструктуры должны находиться в git репозитории, внесение изменений конфигурации напрямую в виртуальных машинах нежелательно
Как проверять: запускать развёртывание и тесты в docker образе ubuntu, все автоматические тесты должны проходить успешно
### Расширенные функции
РФ1. Должна быть настроена система фоновой сборки, развёртывания и тестирования (CI/CD) типа Jenkins (следует также проверить возможности Gitflic CI/CD)

## ТЗ П2. DevSecOps
### Базовые требования
БТ1. Все зависимости должны проходить автоматическую проверку на наличие уязвимостей (проверить CodeQL, SonarLint)
Как проверять: должен быть настроен SAST инструмент, который будет выдавать отчёт после каждого запуска или каждого изменения.
БТ2. Должны быть настроены автоматические проверки перед отправкой изменений в репозиторий (pre-commit hook).
Как проверять: перед отправкой изменений должны автоматически выполняться SAST тесты. В случае ошибок тестирования отправка изменений должна блокироваться
БТ3. Все скрипты (без секретов) для настройки SAST должны находиться в git репозитории, внесение изменений конфигурации напрямую в виртуальных машинах нежелательно
Как проверять: запускать развёртывание и тесты в docker образе ubuntu, все автоматические тесты должны проходить успешно
### Расширенные функции
РФ1. SAST должны быть интегрированы в CI/CD. Допускается выполнение на зеркале репозитория (например, на github) при условии регулярного обновления заркала не реже раза в день
